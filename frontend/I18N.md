# Internationalization

> next-intl-based i18n strategy. English-only at launch, but infrastructure in place for multi-locale expansion. Audit platforms sell globally.

---

## 1. Why i18n From Day One

Retrofitting i18n is expensive. Every hardcoded string becomes a change request. We invest upfront because:

- **Audit is a global market** — customers in EU, LATAM, Asia demand localized UIs
- **Government customers in multi-lingual countries** (Canada: en-CA + fr-CA; Switzerland: de-CH + fr-CH + it-CH; Belgium: nl-BE + fr-BE)
- **Standard packs are authored in English but quoted in local languages** — pack content is translatable
- **Regulatory text must be accurate** — incorrect translation of audit criteria is a compliance risk

Launch in `en-US`. Add `en-GB` before GA (small diff — date/currency). `fr-FR`, `de-DE`, `es-ES` as paying customers request.

## 2. Library — `next-intl`

### Why next-intl over alternatives
| Library | Chose? | Why / Why not |
|---------|--------|---------------|
| **next-intl** ✅ | Yes | Best Next.js 15 App Router support, server + client components, ICU MessageFormat, RSC-safe |
| react-intl | No | Legacy Pages Router-oriented, heavier bundle |
| i18next | No | Works, but more config, less idiomatic for App Router |
| lingui | No | Good DX but smaller ecosystem |
| Next.js built-in i18n | No | Deprecated in App Router |

### Features we use
- ICU MessageFormat (pluralization, selectors, dates, numbers)
- Server-side rendering (no flash of untranslated content)
- Type-safe messages via codegen
- Locale routing (subpath or subdomain)
- Timezone-aware date formatting

## 3. Locale Strategy

### Supported Locales (Phased)

| Locale | Code | Phase | Notes |
|--------|------|-------|-------|
| English (US) | `en-US` | Phase 1 (launch) | Default fallback |
| English (UK) | `en-GB` | Phase 1 | Date format, spelling |
| French (France) | `fr-FR` | Phase 2 | EU market |
| German (Germany) | `de-DE` | Phase 2 | EU market |
| Spanish (Spain) | `es-ES` | Phase 3 | EU + LATAM market |
| Portuguese (Brazil) | `pt-BR` | Phase 3 | LATAM |
| Japanese | `ja-JP` | Phase 4 | Asia-Pacific |
| Arabic (Saudi) | `ar-SA` | Phase 4 | RTL support |
| Hebrew | `he-IL` | Phase 4 | RTL support |

### Locale Resolution (Priority Order)
1. URL path (`/en-US/...`) or subdomain (`fr.aims.io`)
2. User preference (cookie set on selection)
3. `Accept-Language` header (browser)
4. Default (`en-US`)

### URL Strategy
**Subpath routing** at the root: `aims.io/en-US/engagements` vs `aims.io/fr-FR/engagements`.

Why subpath over subdomain:
- Simpler DNS (single cert, no per-locale subdomain)
- Share same session cookie across locales
- Analytics easier (one domain)
- Tenant subdomain (`acme.aims.io`) stacks cleanly: `acme.aims.io/fr-FR/engagements`

## 4. Message Organization

### File Structure
```
apps/web/messages/
├── en-US.json          # Source of truth (English)
├── en-GB.json          # Mostly inherits from en-US, overrides spellings/dates
├── fr-FR.json
├── de-DE.json
└── ...
```

### Message Key Convention
Dot-namespaced, domain-first:

```json
{
  "common": {
    "save": "Save",
    "cancel": "Cancel",
    "delete": "Delete",
    "confirm": "Confirm"
  },
  "engagement": {
    "title": "Engagement",
    "titlePlural": "Engagements",
    "new": "New engagement",
    "list": {
      "title": "Engagements",
      "empty": "No engagements yet",
      "emptyDescription": "Create your first engagement to get started."
    },
    "detail": {
      "tabs": {
        "overview": "Overview",
        "planning": "Planning",
        "fieldwork": "Fieldwork"
      }
    },
    "status": {
      "draft": "Draft",
      "in_progress": "In progress",
      "issued": "Issued",
      "closed": "Closed"
    }
  },
  "finding": { "..." : "..." }
}
```

### Key Naming Rules
- Dot-separated, lowercase segments
- Top-level = feature/domain
- Descriptive leaf (`emptyDescription`, not `e1`)
- Never abbreviate leafs (`description`, not `desc`)
- Never include the default English in the key (`greeting`, not `helloWorld`)

## 5. Usage Patterns

### In Server Components

```tsx
// app/(app)/engagements/page.tsx
import { getTranslations } from "next-intl/server";

export default async function EngagementsPage() {
  const t = await getTranslations("engagement.list");
  return <h1>{t("title")}</h1>;
}
```

### In Client Components

```tsx
"use client";
import { useTranslations } from "next-intl";

export function NewEngagementButton() {
  const t = useTranslations("engagement");
  return <Button>{t("new")}</Button>;
}
```

### ICU Formatting

```json
{
  "findings": {
    "count": "{count, plural, =0 {No findings} =1 {1 finding} other {# findings}}",
    "assignedTo": "Assigned to {name}",
    "lastUpdated": "Last updated {date, date, long}"
  }
}
```

```tsx
t("findings.count", { count: 5 });               // "5 findings"
t("findings.assignedTo", { name: "Alice" });
t("findings.lastUpdated", { date: new Date() });
```

### Rich Text (Inline Formatting)

```tsx
t.rich("termsNotice", {
  link: (chunks) => <Link href="/terms">{chunks}</Link>,
  b: (chunks) => <strong>{chunks}</strong>,
});
// Message: "By continuing you agree to our <link>terms</link>."
```

## 6. Type Safety

### Generated Types
`next-intl` generates TypeScript types from the default locale JSON:

```ts
// messages.d.ts (auto-generated)
type Messages = typeof import("./messages/en-US.json");
declare interface IntlMessages extends Messages {}
```

Effect: `t("engagement.nonExistent.key")` is a **compile error**. Renaming a key breaks builds until all usages are updated.

### Runtime Validation (CI)
A CI step validates every locale file:
- All keys from `en-US` exist in every other locale (no missing translations)
- No extra keys (orphan translations)
- ICU syntax valid
- Same variables used across locales

```bash
pnpm i18n:validate
```

## 7. Date, Number, Currency Formatting

Use `next-intl`'s locale-aware formatters — never string-concatenate dates.

```tsx
const t = await getTranslations();
const format = useFormatter();   // client
// or await getFormatter() in server components

format.dateTime(date, { dateStyle: "long" });
// en-US: "April 19, 2026"
// fr-FR: "19 avril 2026"
// ja-JP: "2026年4月19日"

format.number(amount, { style: "currency", currency: "USD" });
// en-US: "$1,234.56"
// de-DE: "1.234,56 $"

format.relativeTime(date);      // "2 hours ago"
```

### Timezone
User timezone stored in profile; formatting honors it:
```tsx
format.dateTime(date, { dateStyle: "short", timeZone: user.timezone });
```

### Currency per Tenant
Tenants configure reporting currency. All monetary display routes through the tenant currency preference.

## 8. RTL (Right-to-Left) Support

### Required for
Arabic, Hebrew (Phase 4+).

### Implementation
- `<html dir="rtl">` set when locale is RTL
- Tailwind logical properties: `me-2` (margin-end) not `mr-2`, `ps-4` not `pl-4`
- Icons that imply direction (arrow-right) flip via CSS `[dir="rtl"] .icon-arrow { transform: scaleX(-1); }`
- Text alignment follows document direction (never hardcoded `text-left`)
- Number input remains LTR even in RTL contexts
- Dates/times remain in locale format (not reversed)

### Tailwind Logical Utilities
Tailwind v4 supports `ms-`, `me-`, `ps-`, `pe-`, `start-`, `end-` natively. We use these **everywhere**, even for English launch. Means RTL just works when we flip `dir`.

### Component Audit
Every custom component tested in RTL via Storybook + visual regression. Sidebar, tab nav, breadcrumbs, form layouts — all mirror correctly.

## 9. Translating Dynamic / User-Generated Content

### Category 1: UI Strings
Fully translated. Via `messages/<locale>.json`.

### Category 2: Standard Pack Content
Pack definitions include translations:

```ts
{
  code: "criteria",
  label: {
    "en-US": "Criteria",
    "fr-FR": "Critères",
    "de-DE": "Kriterium"
  },
  helpText: {
    "en-US": "The standard against which the condition is measured",
    "fr-FR": "La norme..."
  }
}
```

UI falls back to `en-US` if locale missing. Tenants can add translations via admin panel.

### Category 3: User-Generated Content
Never auto-translated. Finding titles, descriptions, comments remain in the language the author wrote them. Reviewer can request translation; system uses an on-demand translation service (e.g., DeepL, Google Translate API) — but only on explicit request, shown clearly as machine-translated, original preserved.

### Category 4: Reports (PDF)
PDFs respect engagement locale. Templates have translations for section headings ("Executive Summary", "Findings", "Recommendations"). User-entered body text stays in original language.

## 10. Translator Workflow

### Source of Truth
`messages/en-US.json` — product team maintains this.

### Translation Platform (Phase 2+)
Integrate with **Crowdin** or **Lokalise**:
- Auto-sync from `en-US.json` → translation queue
- Translators work in web UI
- Context notes per key (screenshots, description)
- Approved translations sync back → pull request
- Never edit non-English locale files by hand in repo

### Translator Context Notes
Each message can have a `_comment` field (stripped at build):
```json
{
  "engagement": {
    "list": {
      "empty": "No engagements yet",
      "_comment_empty": "Shown on the engagement list page when the tenant has never created any. Friendly tone."
    }
  }
}
```

### Screenshots Per Key
Crowdin/Lokalise attach screenshots automatically on translator request — integrated with Playwright visual capture.

## 11. Pluralization

Beyond English's two forms (singular, plural), use ICU `plural` selector:

```json
{
  "findingsFound": "{count, plural, =0 {No findings} =1 {1 finding} other {# findings}}"
}
```

Languages with more forms (Russian, Arabic, Polish) get additional categories:

```json
{
  "findingsFound": "{count, plural, =0 {...} one {...} few {...} many {...} other {...}}"
}
```

ICU handles selection per-locale automatically.

## 12. Gender / Grammatical Agreement

For languages with grammatical gender (French, Spanish, German):

```json
{
  "assignedTo": "{gender, select, male {Assigné à {name}} female {Assignée à {name}} other {Assigné(e) à {name}}}"
}
```

User profile includes optional gender preference. Fallback: neutral form.

## 13. String Extraction Discipline

### The One Rule
**No hardcoded user-visible strings in code.** Everything goes through `t()`.

Exceptions:
- Log messages (developer-only; English)
- Error messages thrown internally (converted to i18n strings at UI boundary)
- Brand names ("AIMS", "Claude" — proper nouns)

### ESLint Rule
Custom lint rule flags hardcoded strings in JSX/TSX:
```tsx
<Button>Save</Button>          // ❌ ESLint error
<Button>{t("common.save")}</Button>  // ✅
```

Opt-out via comment for genuinely non-translatable (brand): `{/* eslint-disable-next-line i18n/no-literal-string */}`.

## 14. Runtime Locale Switching

User selects locale → cookie set → router pushes to new locale path → page re-renders in new language.

```tsx
"use client";
import { useRouter, usePathname } from "@/i18n/navigation";
import { locales } from "@/i18n/config";

export function LocaleSwitcher() {
  const router = useRouter();
  const pathname = usePathname();
  const currentLocale = useLocale();

  return (
    <Select value={currentLocale} onValueChange={(locale) => router.replace(pathname, { locale })}>
      {locales.map((l) => <SelectItem key={l} value={l}>{localeLabel(l)}</SelectItem>)}
    </Select>
  );
}
```

No page reload — Server Components re-render with new translations.

## 15. Fallback Behavior

### Missing Translation
1. Look in exact locale (`fr-FR`)
2. Fall back to language-only (`fr`)
3. Fall back to default (`en-US`)
4. Last resort: render the key itself (in dev, throw; in prod, log warning)

### Missing Variables
ICU throws if required variables missing. Build-time validation catches these before shipping.

## 16. SEO & i18n

### `hreflang` Tags
Each page emits alternate language links:
```html
<link rel="alternate" hreflang="en-US" href="https://aims.io/en-US/engagements" />
<link rel="alternate" hreflang="fr-FR" href="https://aims.io/fr-FR/engagements" />
<link rel="alternate" hreflang="x-default" href="https://aims.io/engagements" />
```

### Localized Metadata
Page titles, meta descriptions, Open Graph — all translated via `generateMetadata()` using `getTranslations()`.

## 17. Testing i18n

### Unit / Component
- Wrap components in `NextIntlClientProvider` with mock messages in tests
- Test pluralization edge cases (0, 1, 2, many)
- Test gender/locale variants where applicable

### E2E
- Playwright runs critical flows against `fr-FR` locale (at minimum) — verifies no untranslated strings leak
- Pseudolocalization run (`en-XA` — bracketed, elongated "[ŞȧṽḛȱȰȱȰ]") exposes hardcoded strings and truncation bugs

### Visual Regression
- Chromatic/visual tests across locales for components likely to break (dates, long German compounds, RTL layouts)

## 18. Accessibility + i18n

- `<html lang>` set to current locale (screen readers pick up pronunciation)
- `<html dir>` set to `ltr` or `rtl`
- Date formats native to locale (screen readers read correctly)
- Number formats native to locale

See `ACCESSIBILITY.md §14`.

---

## 19. What We Do Not Support

- **Auto-translation of content** — accuracy-critical domain; we ship human translations only
- **User-authored translations at tenant level for UI strings** — tenants can't override built-in UI language (too risky for support)
- **Right-left-right (bidi) mixed text** — basic bidi is fine (Radix handles); complex bidi not prioritized
