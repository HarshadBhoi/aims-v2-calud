# 3.7 — Migration from AIMS v1

> **⚠ Reconciliation 2026-05-01**: DEFERRED from Slice A (covered by Phase 3 OVERVIEW banner). When this resumes, two ADRs constrain the v1 → v2 mapping for control + risk artifacts: [ADR-0008](../../references/adr/0008-control-matrix-as-separate-model.md) maps `Claude_PRCM` rows 1:1 onto a separate `ControlMatrix` table (not collapsed into `AuditTest`); [ADR-0009](../../references/adr/0009-risk-assessment-history-table.md) maps `Claude_RiskAssessments` rows 1:1 onto a `RiskAssessment` per-(entity, FY) history table (not a JSONB blob on `AuditUniverseEntity`). v1's per-FY rows preserve cleanly under both decisions.

**Parent**: [Phase 3 — GAGAS Standard Pack](OVERVIEW.md)
**Estimated effort**: 3-4 days

---

## Objective

Build a migration tool that moves data from AIMS v1 (SharePoint lists) to AIMS v2 (PostgreSQL). Existing customers should not lose any data.

- Test with AIMS v1 sample data before running on production SP data
- Keep migration scripts even after all customers migrate — new customers may come from SP
- Consider a "read-only SP connection" mode during migration period (use v2, reference v1)
