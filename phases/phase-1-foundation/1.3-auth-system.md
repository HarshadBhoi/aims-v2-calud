## Objective

Implement secure authentication (who are you?) and authorization (what can you do?) with support for multi-tenancy, role-based access, and future enterprise SSO.

---

## Tasks

### 1.3.1 — Authentication Setup
- [ ] Install and configure auth library (Better Auth or Auth.js v5)
- [ ] Implement authentication strategies:
  - **Email + Password** (primary, with bcrypt hashing)
  - **Magic Link** (passwordless login via email)
  - **OAuth** (Google, Microsoft — optional for Phase 1)
- [ ] Implement auth endpoints:
  - `POST /auth/register` — Create account (with tenant provisioning)
  - `POST /auth/login` — Email + password login
  - `POST /auth/logout` — Invalidate session
  - `POST /auth/forgot-password` — Send reset email
  - `POST /auth/reset-password` — Reset with token
  - `POST /auth/verify-email` — Email verification
  - `GET /auth/me` — Current user profile
- [ ] JWT tokens (access + refresh token rotation)
- [ ] Secure cookie-based session management (httpOnly, secure, sameSite)
- [ ] Rate limiting on auth endpoints (prevent brute force)

### 1.3.2 — Multi-Factor Authentication (MFA)

- [ ] MFA enrollment flow (QR code + backup codes)
- [ ] MFA verification on login
- [ ] Admin ability to enforce MFA per tenant

### 1.3.4 — Tenant-Scoped Auth
- [ ] Login resolves user to correct tenant (via email domain or tenant selection)
- [ ] JWT includes `tenantId` claim
- [ ] All API requests scoped to user's tenant automatically
- [ ] Tenant admin can invite users (generate invitation link/email)
- [ ] User can belong to multiple tenants (future: tenant switcher)

### 1.3.5 — Auth Pages (Frontend)
- [ ] Login page (email + password, "Remember me", forgot password link)
- [ ] Register page (name, email, password, organization name for new tenant)
- [ ] Forgot password page
- [ ] Reset password page (with token from email)
- [ ] Email verification page
- [ ] MFA setup page
- [ ] MFA challenge page
- [ ] All pages: responsive, accessible, branded with AIMS logo

### 1.3.6 — Security Hardening
- [ ] Password requirements: min 12 chars, complexity check (zxcvbn)
- [ ] Account lockout after 5 failed attempts (15-minute lockout)
- [ ] Session timeout (configurable per tenant, default 30 minutes)
- [ ] CSRF protection
- [ ] Secure headers (helmet)
- [ ] Audit log: all auth events (login, logout, failed login, password change, MFA events)

## Notes

- NEVER store passwords in plain text — bcrypt with salt rounds ≥ 12
- Refresh tokens stored in httpOnly cookies, access tokens short-lived (15 min)
- All auth events written to AuditLog for compliance
- SSO (SAML/OIDC) deferred to Phase 7.3 — enterprise feature
