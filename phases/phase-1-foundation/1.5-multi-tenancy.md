# 1.5 — Multi-Tenancy Architecture

**Parent**: [Phase 1 — Foundation](OVERVIEW.md)
**Estimated effort**: 2-3 days

---

## Objective

Implement multi-tenancy so multiple organizations can use the platform with complete data isolation, independent configuration, and separate billing.

---

## Tasks

### 1.5.1 — Tenancy Strategy

**Approach**: Shared database with Row-Level Security (RLS)

| Strategy | Pros | Cons | When to use |
|----------|------|------|-------------|
| Shared DB + RLS | Cost-effective, easy maintenance | Less isolation | SaaS tier (most customers) |
| Schema-per-tenant | Good isolation, shared infra | Migration complexity | Mid-tier |
| Database-per-tenant | Maximum isolation | Expensive, complex ops | Enterprise / on-prem |

**Implementation**: Start with shared DB + RLS. Add database-per-tenant as an enterprise option later.

### 1.5.2 — Tenant Provisioning
- [ ] Tenant creation flow:
  1. User registers → new Tenant created
  2. User becomes ADMIN of that tenant
  3. Default settings applied
  4. Default standard (GAGAS) activated
  5. Welcome email sent
- [ ] Tenant settings:

- [ ] Tenant slug for URL routing (`app.aims.com/city-of-austin` or subdomain `city-of-austin.aims.com`)

### 1.5.3 — Row-Level Security (RLS)
- [ ] PostgreSQL RLS policies:
  ```sql
  -- Enable RLS on all tenant-scoped tables
  ALTER TABLE engagements ENABLE ROW LEVEL SECURITY;
  
  -- Policy: users can only see their tenant's data
  CREATE POLICY tenant_isolation ON engagements
    USING (tenant_id = current_setting('app.current_tenant')::text);
  
  -- Set tenant context per request
  SET app.current_tenant = 'tenant_cuid_here';
  ```
- [ ] Prisma middleware to automatically:
  - Set `app.current_tenant` before every query
  - Add `tenantId` filter to all reads
  - Set `tenantId` on all creates
- [ ] Test: verify cross-tenant data leakage is impossible

## Data Isolation Verification

```
Test Matrix:
✓ Tenant A admin cannot list Tenant B engagements
✓ Tenant A user cannot access Tenant B finding by ID
✓ Tenant A search results never include Tenant B data
✓ API endpoint with Tenant A JWT returns 404 for Tenant B resources
✓ File storage: Tenant A cannot access Tenant B workpapers
✓ Audit log: Tenant A log never shows Tenant B events
```

---

## Notes

- `tenantId` is NEVER optional on tenant-scoped tables — enforced at Prisma schema level
- Superadmin role (platform-level, not tenant-level) is separate — for platform operators only
- Tenant deletion = soft delete (mark inactive, retain data for compliance period)
