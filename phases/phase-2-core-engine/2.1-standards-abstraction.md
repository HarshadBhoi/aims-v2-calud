## Objective

Design and implement the abstraction layer that allows the audit engine to operate with any audit standard. This is the most architecturally critical piece of the entire platform.

---

## Concept

A **Standard Pack** is a self-contained configuration module that defines:
- **Terminology** — What things are called (e.g., "Finding" vs "Observation" vs "Issue")
- **Required fields** — What data must be collected at each stage
- **Engagement types** — What kinds of audits exist under this standard
- **Phases & workflow** — What stages an engagement goes through
- **Checklists** — Quality control, independence, peer review items
- **Templates** — Planning memo, work program, report structures
- **Report format** — How the final output looks (PDF layout, sections)
- **Crosswalk mappings** — How this standard's requirements map to other standards

## Tasks

### 2.1.1 — Standard Pack Interface

```typescript
// packages/standard-packs/types.ts

interface StandardPack {
  meta: StandardMeta;
  terminology: StandardTerminology;
  engagementTypes: EngagementTypeDefinition[];
  phases: PhaseDefinition[];
  findingElements: FindingElementDefinition[];
  checklists: ChecklistDefinition[];
  templates: TemplateDefinition[];
  reportSections: ReportSectionDefinition[];
  validationRules: ValidationRule[];
  approvalWorkflows: WorkflowDefinition[];
  crosswalkMappings: CrosswalkMapping[];
}

interface StandardMeta {
  code: string;           // "GAGAS"
  name: string;           // "Generally Accepted Government Auditing Standards"
  version: string;        // "2024 Revision"
  issuingBody: string;    // "U.S. Government Accountability Office"
  description: string;
  iconUrl?: string;
}

interface StandardTerminology {
  engagement: string;     // "Engagement" | "Audit" | "Assessment"
  finding: string;        // "Finding" | "Observation" | "Issue" | "Exception"
  recommendation: string; // "Recommendation" | "Suggested Action"
  workpaper: string;      // "Workpaper" | "Working Paper" | "Evidence"
  auditee: string;        // "Auditee" | "Client" | "Auditee Entity"
  fieldwork: string;      // "Fieldwork" | "Testing" | "Examination"
  // ... extensible
}

interface EngagementTypeDefinition {
  code: string;           // "FINANCIAL_AUDIT"
  label: string;          // "Financial Audit"
  description: string;
  applicablePhases: string[];  // Which phases apply to this type
  requiredDocuments: string[]; // What must be produced
  defaultWorkflowCode: string;
}

interface PhaseDefinition {
  code: string;           // "PLANNING"
  label: string;          // "Planning"
  order: number;
  requiredFields: FieldDefinition[];
  requiredDocuments: string[];
  exitCriteria: string[]; // What must be complete before moving to next phase
}

interface FindingElementDefinition {
  code: string;           // "CRITERIA"
  label: string;          // "Criteria"
  description: string;    // "What should be (the standard, regulation, or expectation)"
  isRequired: boolean;
  fieldType: 'text' | 'richtext' | 'select' | 'multi-select';
  helpText: string;       // Guidance for the auditor
  referenceSection: string; // "§6.39" — where this comes from in the standard
}

interface ChecklistDefinition {
  code: string;           // "QA_REVIEW"
  label: string;
  category: string;       // "Quality Assurance" | "Independence" | "Peer Review"
  items: ChecklistItemDefinition[];
}

interface ChecklistItemDefinition {
  id: string;
  text: string;
  referenceSection: string;
  isRequired: boolean;
  category: string;
  responseType: 'yes_no_na' | 'rating' | 'text' | 'checkbox';
}

interface WorkflowDefinition {
  code: string;           // "ENGAGEMENT_APPROVAL"
  label: string;
  entityType: string;     // "engagement" | "finding" | "report"
  steps: WorkflowStepDefinition[];
}

interface WorkflowStepDefinition {
  order: number;
  label: string;          // "Supervisor Review"
  requiredRole: string;   // "SUPERVISOR"
  isOptional: boolean;
  autoAdvanceOnApproval: boolean;
}

interface ReportSectionDefinition {
  code: string;           // "EXECUTIVE_SUMMARY"
  label: string;
  order: number;
  isRequired: boolean;
  description: string;
  fieldType: 'richtext' | 'auto-generated' | 'template';
  referenceSection: string;
}

interface TemplateDefinition {
  code: string;           // "PLANNING_MEMO"
  label: string;
  type: 'planning' | 'fieldwork' | 'reporting';
  sections: TemplateSectionDefinition[];
}

interface CrosswalkMapping {
  thisSection: string;    // "§6.39"
  otherStandardCode: string; // "IIA_GIAS"
  otherSection: string;   // "Standard 2320"
  mappingType: 'equivalent' | 'partial' | 'related';
  notes: string;
}
```

### 2.1.2 — Standard Pack Loader
- [ ] Create `StandardPackService` in NestJS:
  - `getAvailableStandards()` — List all installed standard packs
  - `getStandard(code)` — Load a specific standard pack
  - `getTenantStandards(tenantId)` — Standards activated for a tenant
  - `getEngagementStandard(engagementId)` — Standard(s) for an engagement
- [ ] Standard packs loaded from `packages/standard-packs/{code}/`
- [ ] Each pack is a TypeScript module exporting `StandardPack`
- [ ] Packs can be extended/overridden per tenant (custom checklists, etc.)

### 2.1.3 — Standard-Driven Form Rendering
- [ ] Frontend `StandardContext` provider
- [ ] `useStandard()` hook returning current engagement's standard pack
- [ ] Dynamic form rendering based on `requiredFields` and `findingElements`
- [ ] Field labels, help text, and validation from standard pack (not hardcoded)
- [ ] Example: Finding form renders 4 elements for GAGAS, 3 for IIA, 5 for SOX

### 2.1.4 — Standard-Driven Workflows
- [ ] Approval engine reads workflow definition from standard pack
- [ ] Same approval UI, different steps per standard
- [ ] Workflow steps can vary by engagement type within the same standard

### 2.1.5 — Terminology Engine
- [ ] All UI labels read from `StandardTerminology`
- [ ] `useTerminology()` hook: `const t = useTerminology(); <h1>{t.finding} Detail</h1>`
- [ ] Backend API responses include terminology for the active standard
- [ ] Fallback to English defaults if term not defined

### 2.1.6 — Standard Pack Validation
- [ ] Schema validation for standard packs (ensure all required fields present)
- [ ] Unit tests for each standard pack
- [ ] Runtime validation when loading a pack

## How It Works End-to-End

```
1. User creates engagement → selects Standard (e.g., GAGAS)
2. System loads GAGAS StandardPack
3. Planning form shows GAGAS-specific fields and templates
4. Finding form renders §6.39 four elements
5. QA checklist loads 60+ GAGAS-specific items
6. Approval workflow follows GAGAS review chain
7. Report generates with GAGAS-required sections
8. Crosswalk shows equivalent IIA/SOX requirements
```
