# 2.5 — Findings & Recommendations

**Parent**: [Phase 2 — Core Audit Engine](OVERVIEW.md)
**Estimated effort**: 4-5 days

---

## Objective

Build the findings and recommendations module. Findings are the core audit output — they must be flexible enough to support different standards' requirements while maintaining consistent data integrity.

---

## Tasks

### 2.5.1 — Finding CRUD API
- [ ] tRPC router: `finding.*`
  - `finding.list` — Filterable list (by engagement, status, risk, type, standard)
  - `finding.getById` — Full finding with recommendations, evidence, approvals
  - `finding.create` — Create with standard-driven validation
  - `finding.update` — Update with permission check
  - `finding.delete` — Soft delete (Director+ only)
  - `finding.submitForReview` — Triggers approval workflow
- [ ] Cross-engagement findings register (all findings across all engagements)

### 2.5.2 — Finding Form (Standard-Driven)
- [ ] Dynamic form rendered from `StandardPack.findingElements`:
  - **GAGAS** renders: Criteria, Condition, Cause, Effect (§6.39)
  - **IIA GIAS 2024** renders: Criteria, Condition, Root Cause, Risk/Effect, Recommendation
  - **SOX** renders: Control Deficiency, Impact, Severity Classification
  - **ISO 19011** renders: Nonconformity, Evidence, Standard Clause, Corrective Action
- [ ] Common fields (all standards):
  - Title, finding number, status, risk rating, type/category
  - Background/context, audit response
  - Supporting evidence (linked workpapers)
- [ ] Rich text editor for narrative elements
- [ ] Auto-save drafts
- [ ] Inline validation based on standard requirements

### 2.5.3 — Finding Detail Page
- [ ] Header: finding number, title, status badge, risk rating
- [ ] Standard-specific elements (rendered dynamically)
- [ ] Recommendations panel (linked recommendations)
- [ ] Evidence/workpapers panel
- [ ] Approval status timeline
- [ ] Management response section
- [ ] History/audit trail
- [ ] Actions: Edit, Submit for Review, Approve/Reject, Close

### 2.5.4 — Findings Register (Cross-Engagement)
- [ ] Global view of all findings across all engagements
- [ ] Filters: engagement, status, risk rating, type, standard, date range, auditor
- [ ] Stat cards: Total, by risk rating, by status, overdue follow-ups
- [ ] Trend chart: findings over time, by category
- [ ] Export to Excel/CSV
- [ ] Bulk status update

### 2.5.5 — Recommendation CRUD
- [ ] tRPC router: `recommendation.*`
- [ ] Fields:
  - Title, description, priority
  - Linked finding(s) — one recommendation can address multiple findings
  - Management response (agree/disagree/partial + comments)
  - Agreed action plan
  - Target date
  - Responsible person (can be an auditee)
  - Status: Open → In Progress → Implemented → Verified → Closed
- [ ] Recommendation list page with filters
- [ ] Management response workflow:
  1. Auditor creates recommendation
  2. Auditee receives notification
  3. Auditee submits response (agree/disagree + action plan + target date)
  4. Auditor reviews response
  5. Track implementation and verification

### 2.5.6 — Corrective Action Plans (CAPs)
- [ ] tRPC router: `correctiveAction.*`
- [ ] Fields:
  - Linked recommendation
  - Title, description
  - Status: Not Started → In Progress → Completed → Verified → Overdue
  - Due date, completion date
  - Evidence of completion (file upload)
  - Verified by (auditor), verified date
- [ ] Timeline step view (visual pipeline)
- [ ] Overdue alerts and notifications
- [ ] Follow-up engagement integration

### 2.5.7 — Finding Risk Rating System
- [ ] Configurable risk ratings per standard:
  - GAGAS: Critical, High, Medium, Low, Informational
  - SOX: Material Weakness, Significant Deficiency, Deficiency
  - ISO: Major Nonconformity, Minor Nonconformity, Observation
- [ ] Color-coded badges and charts
- [ ] Risk-based prioritization in lists

---

## Notes

- The finding elements are the strongest test of the standards abstraction layer
- Management response workflow is a key differentiator — auditees interact directly
- Findings flow into reports (Phase 6) — the report auto-pulls findings data
- Cross-standard findings: a single finding can be tagged to multiple standards (crosswalk)
