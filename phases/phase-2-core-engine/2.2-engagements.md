# 2.2 — Engagement Management

**Parent**: [Phase 2 — Core Audit Engine](OVERVIEW.md)
**Estimated effort**: 4-5 days

---

## Objective

Build the full engagement lifecycle — from creation through closure. The engagement is the central entity: everything (findings, workpapers, reports, time) belongs to an engagement.

---

## Tasks

### 2.2.1 — Engagement CRUD API
- [ ] tRPC router: `engagement.*`
  - `engagement.list` — Paginated list with filters (status, type, standard, date range, assignee)
  - `engagement.getById` — Full engagement with relations
  - `engagement.create` — Create with validation
  - `engagement.update` — Update fields (permission-checked)
  - `engagement.delete` — Soft delete (Director+ only)
  - `engagement.clone` — Clone from previous engagement (same type/standard)
  - `engagement.updateStatus` — Transition status with validation
- [ ] Input validation with shared Zod schemas
- [ ] Audit log on every mutation

### 2.2.2 — Engagement Creation Wizard
- [ ] Multi-step wizard UI:
  1. **Type & Standard** — Select engagement type + applicable standard(s)
  2. **Basic Info** — Title, engagement number (auto-generated), description
  3. **Scope** — Objectives, scope, methodology (pre-filled from standard template)
  4. **Timeline** — Planned start/end dates, milestones
  5. **Team** — Assign team members with roles
  6. **Review & Create** — Summary, confirm, create
- [ ] "Clone from previous" option on step 1
- [ ] Standard pack drives available types and required fields
- [ ] Save as draft at any step

### 2.2.3 — Engagement List Page
- [ ] Data table with columns: Number, Title, Type, Standard, Status, Lead, Dates, Risk
- [ ] Filters: Status, Type, Standard, Date Range, Lead Auditor, Risk Rating
- [ ] Search: Title, engagement number
- [ ] Sort by any column
- [ ] Stat cards at top: Total, By Status (Planning, Fieldwork, Reporting, etc.)
- [ ] Card view toggle (grid of cards vs table)
- [ ] Quick actions: View, Edit, Clone

### 2.2.4 — Engagement Detail Page
- [ ] Tabbed interface (carry over pattern from AIMS v1):
  - **Overview** — Status, timeline, scope, risk rating, approval status
  - **Team** — Team members, roles, add/remove
  - **Planning** — Planning memo, risk assessment, work programs
  - **Fieldwork** — Observations, workpapers
  - **Findings** — Findings list, create finding
  - **Recommendations** — Linked recommendations
  - **Reports** — Draft/final reports
  - **Time** — Time entries for this engagement
  - **QA** — QA checklist, peer review
  - **History** — Audit trail for this engagement
- [ ] Status banner showing current phase and next required action
- [ ] Action buttons change based on status and user role

### 2.2.5 — Engagement Status Machine
- [ ] State transitions with validation:
  ```
  DRAFT → PLANNING → FIELDWORK → REPORTING → REVIEW → ISSUED → CLOSED
                                                         ↓
                                                      CANCELLED
  ```
- [ ] Transition rules:
  - DRAFT → PLANNING: requires team assigned
  - PLANNING → FIELDWORK: requires planning doc approved
  - FIELDWORK → REPORTING: requires all work programs complete
  - REPORTING → REVIEW: requires draft report
  - REVIEW → ISSUED: requires approval workflow complete
  - Any → CANCELLED: Director+ only, requires reason
- [ ] Standard pack can override/extend transition rules

### 2.2.6 — Engagement Team Management
- [ ] Add/remove team members
- [ ] Assign roles: Lead Auditor, Staff Auditor, Supervisor, QA Reviewer
- [ ] Restrict access: only team members can see engagement workpapers
- [ ] Notifications when added to/removed from team
- [ ] Capacity view: see each person's current engagements

### 2.2.7 — Engagement Cloning
- [ ] Clone engagement from a previous one (common for annual audits)
- [ ] Carries over: scope, methodology, work programs, team template, standard
- [ ] Does NOT carry over: findings, workpapers, reports, time entries
- [ ] Auto-increment engagement number
- [ ] Update dates to new period

---

## Notes

- Engagement number format configurable per tenant (e.g., "ENG-2026-001")
- Engagement is always scoped to a tenant (tenantId in all queries)
- The engagement detail page is the "home base" — auditors live here during an audit
